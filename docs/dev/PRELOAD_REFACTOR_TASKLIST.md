# 预加载重构任务清单
> 更新日期：2026-04-14
>
> 配套文档：`docs/dev/PRELOAD_ASSESSMENT.md`

## 0. 2026-04-14 复核修正

本清单需要补一条非常关键的修正：

- 之前 Phase B 更偏向“缓存层是否已经能复用”。
- 但当前用户反馈和代码复核说明，**缓存层复用成立，不等于进入播放器已经明显变快**。
- 最新构建后手测还进一步说明：**后半段已经改善，前半段仍没有改善**。
- 现在看不到的不是“有没有复用缓存”，而是“点击播放到首帧之间的那段等待为什么还在”。
- 根因已经不再只是“结果没有带进来”，而是 handoff 之后仍有少量 cache fill、播放器初始化、首探测请求处在关键路径；其中 EXO 命中 handoff 后重复 `_buildPlaybackSource()` / proxy wrap 这一项已在 2026-04-14 当日后续修正中补齐收口。

因此从现在开始，新的 P0 不是单纯继续补 warmup，而是：

- 把详情页 / 下一集阶段已经得到的 `PreparedPlaybackPreload`、`ResolvedPlaybackSource` 或等价结果，真正带进播放页
- 不再让播放页只拿 `itemId + startPosition + mediaSourceId` 重新全量准备

### 0.1 同日最新手测补记

本轮最新手测还需要再补两条非常关键的现状说明：

- `MPV` 内核下，即使预加载已经完成，点击播放后在真正开始播放前，仍会继续发生**明显偏大的缓存补齐**，而且这段补齐量已经不是“首帧前少量 cache fill”，更接近“仍有一段较重的继续缓存后才开播”。
- `EXO` 内核下，当前片源已经可以正常播放，不再以“播放直接报错”作为主要矛盾；但“预加载完成后命中缓存再开播”这件事仍然**明显偏慢**，说明 Exo 路径的预加载收益还没有顺畅转化为首帧前时延下降。

这意味着现在不能再只写“前半段仍需收口”，而要明确拆成：

- `MPV`：前半段仍有**偏重的继续缓存 / playback fill**，是最突出的体感问题。
- `EXO`：播放可用性已恢复，但“加载预加载缓存”本身仍耗时偏长，问题更偏向**缓存命中后到控制器真正开始播放**这段链路。

### 0.2 2026-04-14 同日最新回归补记（本轮尝试已修烂）

在 2026-04-14 同日后续一轮“播放页 startup preload barrier + loopback fast-start profile”尝试后的最新构建上，用户手测反馈需要把问题定义进一步改写：

- `MPV` 内核下，这轮尝试后仍然需要明显大量缓存才能开始播放，说明前半段重缓存问题**没有被解决**。
- `EXO` 内核下，这轮尝试后不仅没有改善，反而出现了**更严重回归**：只能看到流量一直流出，但播放页 UI 甚至没有出现，用户根本无法开始观看。
- 这说明当前不能再把这轮尝试记成“继续收口前半段”，而必须记成**已引入回归的失败尝试**。
- 也说明把 preload barrier / warmup 等待直接推进播放页关键路径，是高风险做法；在没有定位清楚前，不能继续默认沿这个方向加码。

因此从现在开始，新的最优先事项要先加一条：

- 先恢复 `EXO` 到“能进播放页、能看到 UI、能开始观看”的基线
- 再重新定位 `MPV` 首帧前大量缓存的真实根因
- 在没有证明安全前，不再把“播放页入口同步等待 preload”视为默认优化方向

## 1. 先把“预加载”定义清楚

这轮要做的预加载，不是“后台尽量多下载一点数据”，也不是“为了架构好看引入一个下载器服务”。

这轮真正要实现的预加载，只有下面三类触发场景：

- 用户点进剧集详情页时，提前准备当前集的起播数据
- 用户点进电影详情页时，提前准备当前电影的起播数据
- 当前集即将播放结束、用户大概率会看下一集时，提前准备下一集的起播数据

只要用户点击播放后，播放器能够**直接从这段已经准备好的缓存开始播放**，就算预加载成功。

换句话说：

- 预加载的重点是“提前准备可直接播放的起点”
- 不是“后台多跑一个下载任务”
- 更不是“是否存在一个独立下载器类”

## 2. 这轮真正的成功标准

只有满足下面这些条件，才算“预加载真的有用”：

- 如果预加载已完成起播所需数据，用户点击播放后应直接从这份已准备好的缓存起播
- 等预加载完成后再点播放，起播等待明显下降
- 不会出现“后台持续出流，但播放页 UI 未出现、用户无法观看”这类阻塞级回归
- 不会重新下载已经预加载完成的起播前缀
- 离开当前内容后，不再相关的后台预加载会及时停止
- 已经落地的缓存可以保留，但不会继续无意义地下载旧内容
- MPV 和 EXO 的行为方向一致

对不同类型片源，这里的“起播数据”含义不同：

- direct file / redirect file / body-link file：起播附近的最小必要字节区间
- HLS：起播所需的 playlist / init / 前几个 segment

## 3. 这轮不把什么当目标

下面这些东西可以是实现手段，但都不是本轮的验收目标：

- 不把“独立预加载下载器服务”当作目标
- 不把“后台尽量缓存更多数据”当作目标
- 不把“完整下载器产品化 UI”当作目标
- 不把“Web 端支持”当作目标
- 不把“转码 / ABR / 自适应码率可观测性”当作目标
- 不把“HLS 复杂 ABR 场景精确命中”当作目标

本项目当前对 HLS 的边界是：

- 仅处理直接流式传输的 HLS
- 不考虑转码
- 不把复杂 ABR 选择作为本轮问题

## 4. 当前真实问题

经过最近的真实手测与代码复核，已经能明确看到当前问题不在“有没有触发预加载”，而在下面这三类闭环还没有真正成立。

### 4.1 起播复用闭环还没有完全成立

当前现象：

- 进入播放后仍然需要等待加载
- 即使等预加载跑完再点播放，体感上仍然没有做到“直接播”
- 最新构建后手测里，这段等待期间仍会补少量缓存，然后才真正开始播放
- 某些场景下仍然像是重新走了一次起播链路
- `MPV` 最新手测中，这段等待期间仍会继续缓存**明显较大一段内容**后才真正开始播放
- `EXO` 最新手测中，虽然已能正常播放，但“加载预加载缓存”本身仍需要明显时间，仍未达到“点开即吃到已预热缓存”的体感

这说明：

- 预加载拿到的数据，还没有被播放器以稳定方式直接消费
- 或者虽然写进了共享缓存，但还没有准确覆盖播放器起播真正最先需要的那部分数据
- 或者播放器起播前仍然存在额外的探测 / 元数据 / 初始化等待
- 现在更准确地说，缓存复用和后半段补流已经部分成立，但点击播放到首帧前的准备等待仍未收口
- 并且现在已经可以进一步拆开看：`MPV` 更像是“首帧前继续补流量过重”，`EXO` 更像是“首帧前消费已预热缓存的链路仍偏慢”

### 4.2 任务生命周期闭环还没有成立

当前现象：

- 退出播放页后，旧片源仍然继续缓存
- 即使切到另一个剧 / 电影，旧片源预加载也可能没有及时停止

这说明：

- 当前任务能启动，但缺少明确的 owner / scope / cancel 机制
- “缓存文件可以保留” 和 “后台下载必须停止” 还没有被清楚地区分
- 预加载还没有真正纳入页面级 / 播放会话级生命周期

### 4.3 播放入口准备链路仍然是重新计算的

当前现象：

- 详情页已经完成预加载，点击播放后仍然要等播放页自己准备
- 下一集预加载完成后，切到下一集仍然不像“拿现成结果直接开播”
- 进入播放页的慢，和进入后继续缓冲，常常被混在一起

这说明：

- 预加载结果已经通过 handoff 带进播放页，但目前主要收口在“共享缓存层 + prepared source 层”
- 播放页命中 handoff 时虽然不再重做 `_buildResolvedPlaybackSource()`，但剩余关键路径仍有播放器初始化、首探测请求、少量 cache fill；其中 MPV / EXO 对已附带 `playbackSource` 的 handoff 现已不再重复 `_buildPlaybackSource()` 与本地代理包装
- 远端续播点拉取已经不再阻塞起播主路径，但这还不足以消除点击播放到首帧前的等待

## 5. 当前的核心判断

### 5.1 预加载下载器不是必须的

之前没有“预加载下载器”这个抽象时，预加载依然可以工作，只是：

- 预加载结果不能稳定被后续播放复用
- 任务生命周期也没有被真正收口

因此，真正必须的不是“下载器服务”，而是下面三件事：

- 能把预加载数据写进共享缓存
- 播放时如果起播数据已就绪，就直接从这同一份缓存开始
- 旧任务在失去相关性后能立即停止

所以这轮的判断应该是：

- `StreamCacheDownloadService` 不再作为独立服务保留
- 保留 `StreamCacheDownloadRequest` 作为共享缓存语义对象
- 是否保留为独立服务，不是验收标准

真正的验收标准只有：

- 起播是否更快
- 前缀是否不重复下载
- 离开当前内容后旧任务是否停止

### 5.2 重点是“可播放起点”，不是“尽量下载更多”

预加载做得再多，如果点击播放时播放器仍然没有先消费这些数据，就没有意义。

因此这轮要盯住的是：

- 起播最小可播集合是否已经就绪
- 播放器首个关键请求是否命中这份缓存
- 下一集预加载是否只做“够起播”的准备，而不是继续无限后台跑

### 5.3 生命周期控制和缓存复用同等重要

如果复用做出来了，但退出播放后旧任务还继续抢带宽，那对用户体验仍然是负收益。

所以这轮不是只有一个目标，而是两个并列的 P0：

- P0：预加载结果能被点击播放直接复用
- P0：旧任务在用户离开当前内容后会及时停止

## 6. 已有基础，哪些东西可以继续复用

下面这些工作已经做出来了，不需要推倒重来：

- [x] `PlaybackSourceBuilder` / `ResolvedPlaybackSource` 已经统一了大部分播放源决策
- [x] 详情页、MPV 播放页、EXO 播放页都已经接入统一预加载入口
- [x] 当前片源 / 下一集 / 续播点的触发链路已经接通
- [x] 本地缓存代理已经具备基础缓存、warmup、range 处理能力
- [x] direct file / redirect file / body-link file 的共享缓存主链路已经打通
- [x] HLS 直传场景的基础缓存链路已经打通
- [x] 自动化测试已经覆盖一批关键缓存复用路径

这些成果说明：

- 现在的问题不是“从零开始做预加载”
- 而是要把“已有能力”真正收口到用户体感和任务生命周期

## 7. 设计原则

接下来所有实现与取舍，都按下面这些原则判断：

- 以“点击播放能否直接用缓存起播”为中心，不以抽象层次为中心
- 以“任务是否可取消、可收口”为中心，不以后台下载是否跑起来为中心
- 以“起播最小可播集合”为中心，不以“尽量多下” 为中心
- 以“共享缓存对象” 为中心，不以“是否存在一个独立下载器服务” 为中心
- 以“手工体感验证” 为最终裁判，不以“自动化测试全绿” 代替真实体验

## 8. 当前优先级

### P0：先止血最新这轮 `EXO` 阻塞级回归

必须先回答清楚：

- 为什么最新尝试后 `EXO` 会出现“流量持续流出，但播放页 UI 未出现”
- 卡住的位置到底在进入播放页前的 preload barrier、播放源包装、代理链路，还是 `controller.initialize()`
- 在恢复到“可进页、可观看”之前，哪些新尝试必须先撤回或隔离，不能继续留在关键路径

在这件事没查清前，不能再把当前工作树视为“只是前半段还没优化完”，而应视为**存在明显回归**。

### P0：把预加载结果直接带进播放页

必须先回答清楚：

- 详情页 / 下一集预加载完成后，播放页到底拿到了什么
- 哪些结果可以直接复用，哪些只能重新计算
- 是否还能继续接受“只传 itemId，进页后全部重算”这种结构

### P0：确认起播等待到底卡在哪里

必须先回答清楚：

- 播放器首个关键请求到底是什么
- 它有没有命中已经准备好的缓存
- 等预加载完成后再点播放，**前半段**等待时间到底消耗在 `_buildPlaybackSource()`、proxy wrap、player initialize、首探测请求还是别的准备动作
- **后半段**已经减少了哪些重复缓存 / 重复回源，哪些补齐仍然发生在首帧前
- `MPV` 路径里，首帧前那段**大体量继续缓存**到底由谁触发：MPV 自身起播策略、loopback proxy `playbackFill`、还是代理对播放器首请求的保护性补齐
- `EXO` 路径里，缓存命中后仍然偏慢，主要耗时是在 `controller.initialize()`、首个探测请求、还是 Exo 对已预热缓存的实际消费节奏

### P0：补上任务 owner / scope / cancel 机制

必须先回答清楚：

- 这个预加载是谁触发的
- 它归属于哪个页面 / 哪个播放会话
- 页面退出、切条目、切集、pushReplacement 时，哪些任务要立即停
- 哪些缓存数据保留，哪些后台下载必须终止

### P1：收口“预加载只做够起播的准备”

必须避免下面这种情况：

- 旧片源还在继续后台跑
- 新片源准备起播时和旧任务抢带宽
- 下一集预加载比当前集真正起播还更重

## 9. 下一阶段任务清单

## Phase A：重新定义预加载的输入与成功判定

目标：把“预加载成功”从“后台下载发生过”改成“点击播放可直接利用缓存起播”。

- [x] 明确三类触发场景：剧集详情页、电影详情页、下一集
- [x] 明确本轮 HLS 范围：仅 direct HLS / 固定源链路 / 非 ABR 目标
- [x] 为 direct file / redirect file / body-link file 定义“起播最小可播集合”：0 秒起播时要求前缀起播窗口已就绪；续播时要求首段探测前缀 + 续播点对应字节窗口都已进入共享缓存
- [x] 为 HLS 定义“起播最小可播集合”：播放入口 playlist（master / media）+ 实际选中的 variant playlist（如有）+ init segment（如有）+ 起播点对应首段 segment（最多 3 段 / 约 3 秒）
- [x] 把预加载成功判定从“下载过”改成“起播可直接利用缓存”
- [x] 为详情页当前项、详情页下一项、播放页当前项、播放页下一项定义 owner / scope

验收标准：

- 每一种触发场景都能清楚回答“预加载到底在准备什么”
- 不再用“有没有下载字节”代替“能不能直接播”

## Phase B：把起播缓存复用真正闭环

目标：让播放器点击播放后，如果起播数据已准备完成，就直接从这份缓存起播；只有超出已准备范围后才继续补齐。

- [x] 预加载和播放已共享 `ResolvedPlaybackSource`
- [x] MPV 播放优先走本地缓存代理 URL
- [x] EXO 播放优先走本地缓存代理 URL
- [x] 当前片源预加载与实际播放已共享同一个缓存对象
- [x] direct file / redirect file / body-link file 的基础复用测试已覆盖
- [x] HLS 直传场景的基础复用测试已覆盖
- [x] 找出“进入播放仍需要等待”的具体耗时点
- [x] 找出“播放器首个关键请求”是否命中缓存
- [x] 从真机诊断确认：播放器已走 loopback cache proxy，但缓存命中后的 `cached-prefix + remote-tail` 在播放器探测断开时被误记为 `proxy-error/cache-serve-error`，会干扰起播复用判断
- [x] 从真机诊断确认：部分续播 / 探测请求仍会出现 `range-not-covered`，需要继续以“播放器实际请求范围”为准校正起播最小集合
- [x] 确认等预加载完成后再点播放，不再重复下载起播前缀
- [x] 确认 direct file 场景下，点击播放后能直接从已缓存起点开始
- [x] 确认 HLS 场景下，点击播放后能直接从已缓存的 playlist / init / 首段 segment 开始
- [x] 详情页 / 下一集预加载完成后，播放页直接复用已准备好的 `PreparedPlaybackPreload` / `ResolvedPlaybackSource`
- [x] 远端续播点拉取已移出播放页起播阻塞主路径，晚到结果改为异步补热 / 自动 seek / 提示
- [x] EXO 播放页命中 handoff 时也已优先复用 `PreparedPlaybackPreload.playbackSource`，不再重复 `_buildPlaybackSource()` / loopback proxy 包装
- [x] 本地 HTTP 代理已支持无 `Range` 的普通 `GET` 等待并复用 in-flight cache fill，避免并发起播重复回源
- [x] 最新构建后手测确认：加载到开始播放的后半段，已基本看不到预加载前缀再次进入缓存流量
- [ ] `EXO` 路径下，不再出现“持续出流但播放页 UI 未出现、无法观看”的严重回归
- [ ] 不再把新的 preload barrier / warmup wait 直接放进播放页 UI / controller initialize 前的阻塞路径，除非先证明它不会引入回归
- [ ] 进入播放页后，不再因为播放器初始化、首探测请求、少量 cache fill 而抵消掉预加载收益
- [ ] 点击播放到首帧前，首帧前的少量 cache fill 也能明显下降
- [ ] `MPV` 路径下，预加载完成后点击播放，不再先继续缓存一大段内容才开始播放
- [ ] `EXO` 路径下，预加载完成后点击播放，“加载预加载缓存”本身的等待要明显缩短，而不只是最终能播放

本轮已完成路由级 handoff、播放页优先复用 prepared result、EXO 命中 handoff 时直复用 prepared `playbackSource`、代理层普通 GET 复用 in-flight cache fill，以及把远端续播点拉取移出起播阻塞主路径；但 2026-04-14 同日最新手测说明，后续一轮把 preload 更进一步推进到播放页关键路径的尝试已经出现明显回归，因此当前阶段不能再简单写成“继续优化前半段”，而要拆成：

- `MPV`：问题重点仍是首帧前偏重的继续缓存，本轮尝试没有解决
- `EXO`：问题重点已升级为播放入口阻塞级回归，必须先恢复到可进页、可看 UI、可开始观看

验收标准：

- `EXO` 先恢复到“不会卡在流量持续流出但播放页 UI 未出现”的基线
- 等预加载完成后再点播放，起播等待明显下降
- 起播前缀不再重复回源
- `MPV` / `EXO` 至少在主流 direct file 场景下都能体现明确收益，而不是一个“继续大量缓存”一个“加载缓存仍很慢”
- 播放页进入阶段不会因为“结果没有带下来”而重新支付完整准备成本
- 点击播放到首帧的前半段等待也能被明确压缩，而不只是后半段补流减少

## Phase C：收口生命周期与取消机制

目标：让预加载任务受页面和播放会话控制，而不是无主后台任务。

- [x] 给预加载任务定义 owner / scope
- [x] 详情页销毁时取消属于该详情页 scope 的任务
- [x] 播放页退出时取消属于该播放会话 scope 的任务
- [x] 切到另一个剧 / 电影时取消不再相关的旧任务
- [x] 切集 / pushReplacement / 重建播放页时取消已失效的旧 scope
- [x] 区分“保留已落地缓存”与“停止继续后台下载”
- [x] 给取消结果补诊断，能看见是正常完成、失败还是被中止
- [x] 确认取消旧任务不会误伤当前新片源的预加载和播放
- [x] 播放页退出时显式取消本地 HTTP 代理中的 active `playbackFill`，避免只取消 preload owner 但播放器补齐下载仍继续跑
- [x] 本地 HTTP 代理把客户端断开 / 主动取消从 `proxy-error` 中区分出来，并使用 abortable request 尽快停止远端流读取

验收标准：

- 退出播放页后，旧片源活动下载会在短时间内归零
- 切到新内容后，不再看到旧片源继续回源下载
- 已缓存数据仍可保留，但旧下载不会继续抢带宽

## Phase D：把“预加载只做够起播”这件事收口

目标：让当前集、续播点、下一集三类预加载都只准备真正有价值的那一小段。

- [x] 当前片源从 0 秒开播时预热前缀区间
- [x] 当前片源从续播点进入时预热续播区间
- [x] 下一集预加载已沿用系列级版本偏好
- [x] 当前集与下一集缓存 key 已隔离
- [x] 详情页预热和播放页兜底预热已共享同一缓存状态
- [x] 限制下一集预加载不要干扰当前集的起播和播放
- [x] 确认剧集详情页 current / next 两类任务不会互相抢带宽
- [x] 确认电影详情页不会在用户离开后继续进行无意义下载
- [x] 确认续播场景命中的是续播点附近真正需要的起播区间

验收标准：

- 三类触发场景都只准备“够起播”的数据
- 当前内容起播优先级始终高于下一集和旧任务

## Phase E：实现手段收口，而不是围着“下载器”打转

目标：把实现手段降级为实现细节，不让它反过来主导产品行为。

- [x] 已有一层共享缓存写入入口
- [x] 评估 `StreamCacheDownloadService` 是否继续保留为独立服务
- [x] 如果该抽象继续增加复杂度，则下沉或内聚回 preload / cache proxy 主链路
- [x] 不再把“是否存在独立下载器”写成任务验收条件
- [x] 所有后续改动都以“缓存复用 + 生命周期控制”为验收口径

当前结论：

- 删除 `StreamCacheDownloadService`
- 保留 `StreamCacheDownloadRequest` 作为缓存语义对象
- preload 主链路直接围绕 `StreamPreloadService` + `HttpStreamProxyServer` + 共享 `CacheKey`

验收标准：

- 文档、代码和调试口径都不再把“下载器存在与否”当成功标准
- 任何实现形态都必须服务于“起播可复用”和“退出即停”

## 12. 完成定义

当且仅当下面这些条件都成立时，才算这轮预加载真正可用：

- [ ] 剧集详情页进入后，当前集点击播放能直接利用缓存起播
- [ ] 电影详情页进入后，当前电影点击播放能直接利用缓存起播
- [ ] 下一集切换时，下一集能直接利用缓存更快起播
- [ ] 续播进入时，续播点附近能直接利用缓存更快起播
- [ ] `EXO` 进入播放页时，不再出现“后台持续缓存 / 出流，但 UI 未出现”的阻塞或假死
- [ ] 进入播放页后，不会因为重新初始化播放器、首探测请求、少量 cache fill 而明显抵消掉预加载收益
- [ ] 预加载完成后再点播放，点击播放到首帧的前半段等待也明显缩短，而不只是后半段更稳

- [ ] 切到另一部剧 / 电影后，旧片源不会继续缓存
- [ ] MPV / EXO 的方向性结论一致

## 13. 最终判断

当前这件事的重点，已经不应该再表述成：

- “要不要一个下载器服务”
- “要不要继续给后台多加点缓存能力”

更准确的说法应该是：

> 我们要把预加载做成一个真正服务于播放起点的能力。  
> 用户在剧集详情页、电影详情页、以及准备看下一集这三个时机里，提前拿到可播放起点；点击播放后不但后半段不再重复灌缓存，前半段到首帧的等待也要明显缩短；离开当前内容后，不再相关的后台任务立即停止。

只要没有做到这件事，无论代码里有多少抽象、多少测试、多少缓存层，都还不算真正完成。
