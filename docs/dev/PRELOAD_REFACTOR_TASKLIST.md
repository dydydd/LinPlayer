# 预加载重构任务清单
> 更新日期：2026-04-06
>
> 配套文档：`docs/dev/PRELOAD_ASSESSMENT.md`

## 1. 先把“预加载”定义清楚

这轮要做的预加载，不是“后台尽量多下载一点数据”，也不是“为了架构好看引入一个下载器服务”。

这轮真正要实现的预加载，只有下面三类触发场景：

- 用户点进剧集详情页时，提前准备当前集的起播数据
- 用户点进电影详情页时，提前准备当前电影的起播数据
- 当前集即将播放结束、用户大概率会看下一集时，提前准备下一集的起播数据

只要用户点击播放后，播放器能够**直接从这段已经准备好的缓存开始播放**，就算预加载成功。

换句话说：

- 预加载的重点是“提前准备可直接播放的起点”
- 不是“后台多跑一个下载任务”
- 更不是“是否存在一个独立下载器类”

## 2. 这轮真正的成功标准

只有满足下面这些条件，才算“预加载真的有用”：

- 用户点击播放后，播放器优先消费已经准备好的同一份缓存
- 等预加载完成后再点播放，起播等待明显下降
- 不会重新下载已经预加载完成的起播前缀
- 离开当前内容后，不再相关的后台预加载会及时停止
- 已经落地的缓存可以保留，但不会继续无意义地下载旧内容
- MPV 和 EXO 的行为方向一致

对不同类型片源，这里的“起播数据”含义不同：

- direct file / redirect file / body-link file：起播附近的最小必要字节区间
- HLS：起播所需的 playlist / init / 前几个 segment

## 3. 这轮不把什么当目标

下面这些东西可以是实现手段，但都不是本轮的验收目标：

- 不把“独立预加载下载器服务”当作目标
- 不把“后台尽量缓存更多数据”当作目标
- 不把“完整下载器产品化 UI”当作目标
- 不把“Web 端支持”当作目标
- 不把“转码 / ABR / 自适应码率可观测性”当作目标
- 不把“HLS 复杂 ABR 场景精确命中”当作目标

本项目当前对 HLS 的边界是：

- 仅处理直接流式传输的 HLS
- 不考虑转码
- 不把复杂 ABR 选择作为本轮问题

## 4. 当前真实问题

经过最近的真实手测，已经能明确看到当前问题不在“有没有触发预加载”，而在下面这两类闭环还没有真正成立。

### 4.1 起播复用闭环还没有完全成立

当前现象：

- 进入播放后仍然需要等待加载
- 即使等预加载跑完再点播放，体感上仍然没有做到“直接播”
- 某些场景下仍然像是重新走了一次起播链路

这说明：

- 预加载拿到的数据，还没有被播放器以稳定方式直接消费
- 或者虽然写进了共享缓存，但还没有准确覆盖播放器起播真正最先需要的那部分数据
- 或者播放器起播前仍然存在额外的探测 / 元数据 / 初始化等待

### 4.2 任务生命周期闭环还没有成立

当前现象：

- 退出播放页后，旧片源仍然继续缓存
- 即使切到另一个剧 / 电影，旧片源预加载也可能没有及时停止

这说明：

- 当前任务能启动，但缺少明确的 owner / scope / cancel 机制
- “缓存文件可以保留” 和 “后台下载必须停止” 还没有被清楚地区分
- 预加载还没有真正纳入页面级 / 播放会话级生命周期

## 5. 当前的核心判断

### 5.1 预加载下载器不是必须的

之前没有“预加载下载器”这个抽象时，预加载依然可以工作，只是：

- 预加载结果不能稳定被后续播放复用
- 任务生命周期也没有被真正收口

因此，真正必须的不是“下载器服务”，而是下面三件事：

- 能把预加载数据写进共享缓存
- 播放时能优先消费这同一份缓存
- 旧任务在失去相关性后能立即停止

所以这轮的判断应该是：

- `StreamCacheDownloadService` 可以保留
- 也可以继续下沉、内聚、弱化存在感
- 是否保留为独立服务，不是验收标准

真正的验收标准只有：

- 起播是否更快
- 前缀是否不重复下载
- 离开当前内容后旧任务是否停止

### 5.2 重点是“可播放起点”，不是“尽量下载更多”

预加载做得再多，如果点击播放时播放器仍然没有先消费这些数据，就没有意义。

因此这轮要盯住的是：

- 起播最小可播集合是否已经就绪
- 播放器首个关键请求是否命中这份缓存
- 下一集预加载是否只做“够起播”的准备，而不是继续无限后台跑

### 5.3 生命周期控制和缓存复用同等重要

如果复用做出来了，但退出播放后旧任务还继续抢带宽，那对用户体验仍然是负收益。

所以这轮不是只有一个目标，而是两个并列的 P0：

- P0：预加载结果能被点击播放直接复用
- P0：旧任务在用户离开当前内容后会及时停止

## 6. 已有基础，哪些东西可以继续复用

下面这些工作已经做出来了，不需要推倒重来：

- [x] `PlaybackSourceBuilder` / `ResolvedPlaybackSource` 已经统一了大部分播放源决策
- [x] 详情页、MPV 播放页、EXO 播放页都已经接入统一预加载入口
- [x] 当前片源 / 下一集 / 续播点的触发链路已经接通
- [x] 本地缓存代理已经具备基础缓存、warmup、range 处理能力
- [x] direct file / redirect file / body-link file 的共享缓存主链路已经打通
- [x] HLS 直传场景的基础缓存链路已经打通
- [x] 自动化测试已经覆盖一批关键缓存复用路径

这些成果说明：

- 现在的问题不是“从零开始做预加载”
- 而是要把“已有能力”真正收口到用户体感和任务生命周期

## 7. 设计原则

接下来所有实现与取舍，都按下面这些原则判断：

- 以“点击播放能否直接用缓存起播”为中心，不以抽象层次为中心
- 以“任务是否可取消、可收口”为中心，不以后台下载是否跑起来为中心
- 以“起播最小可播集合”为中心，不以“尽量多下” 为中心
- 以“共享缓存对象” 为中心，不以“是否存在一个独立下载器服务” 为中心
- 以“手工体感验证” 为最终裁判，不以“自动化测试全绿” 代替真实体验

## 8. 当前优先级

### P0：确认起播等待到底卡在哪里

必须先回答清楚：

- 播放器首个关键请求到底是什么
- 它有没有命中已经准备好的缓存
- 等预加载完成后再点播放，等待时间到底消耗在回源、探测、元数据还是别的准备动作

### P0：补上任务 owner / scope / cancel 机制

必须先回答清楚：

- 这个预加载是谁触发的
- 它归属于哪个页面 / 哪个播放会话
- 页面退出、切条目、切集、pushReplacement 时，哪些任务要立即停
- 哪些缓存数据保留，哪些后台下载必须终止

### P1：收口“预加载只做够起播的准备”

必须避免下面这种情况：

- 旧片源还在继续后台跑
- 新片源准备起播时和旧任务抢带宽
- 下一集预加载比当前集真正起播还更重

## 9. 下一阶段任务清单

## Phase A：重新定义预加载的输入与成功判定

目标：把“预加载成功”从“后台下载发生过”改成“点击播放可直接利用缓存起播”。

- [x] 明确三类触发场景：剧集详情页、电影详情页、下一集
- [x] 明确本轮 HLS 范围：仅 direct HLS / 固定源链路 / 非 ABR 目标
- [ ] 为 direct file / redirect file / body-link file 定义“起播最小可播集合”
- [ ] 为 HLS 定义“起播最小可播集合”
- [ ] 把预加载成功判定从“下载过”改成“起播可直接利用缓存”
- [ ] 为详情页当前项、详情页下一项、播放页当前项、播放页下一项定义 owner / scope

验收标准：

- 每一种触发场景都能清楚回答“预加载到底在准备什么”
- 不再用“有没有下载字节”代替“能不能直接播”

## Phase B：把起播缓存复用真正闭环

目标：让播放器点击播放后，真正优先消费已经准备好的那一份缓存。

- [x] 预加载和播放已共享 `ResolvedPlaybackSource`
- [x] MPV 播放优先走本地缓存代理 URL
- [x] EXO 播放优先走本地缓存代理 URL
- [x] 当前片源预加载与实际播放已共享同一个缓存对象
- [x] direct file / redirect file / body-link file 的基础复用测试已覆盖
- [x] HLS 直传场景的基础复用测试已覆盖
- [ ] 找出“进入播放仍需要等待”的具体耗时点
- [ ] 找出“播放器首个关键请求”是否命中缓存
- [ ] 确认等预加载完成后再点播放，不再重复下载起播前缀
- [ ] 确认 direct file 场景下，点击播放后能直接从已缓存起点开始
- [ ] 确认 HLS 场景下，点击播放后能直接从已缓存的 playlist / init / 首段 segment 开始

验收标准：

- 等预加载完成后再点播放，起播等待明显下降
- 起播前缀不再重复回源
- MPV / EXO 至少在主流 direct file 场景下结论一致

## Phase C：收口生命周期与取消机制

目标：让预加载任务受页面和播放会话控制，而不是无主后台任务。

- [ ] 给预加载任务定义 owner / scope
- [ ] 详情页销毁时取消属于该详情页 scope 的任务
- [ ] 播放页退出时取消属于该播放会话 scope 的任务
- [ ] 切到另一个剧 / 电影时取消不再相关的旧任务
- [ ] 切集 / pushReplacement / 重建播放页时取消已失效的旧 scope
- [ ] 区分“保留已落地缓存”与“停止继续后台下载”
- [ ] 给取消结果补诊断，能看见是正常完成、失败还是被中止
- [ ] 确认取消旧任务不会误伤当前新片源的预加载和播放

验收标准：

- 退出播放页后，旧片源活动下载会在短时间内归零
- 切到新内容后，不再看到旧片源继续回源下载
- 已缓存数据仍可保留，但旧下载不会继续抢带宽

## Phase D：把“预加载只做够起播”这件事收口

目标：让当前集、续播点、下一集三类预加载都只准备真正有价值的那一小段。

- [x] 当前片源从 0 秒开播时预热前缀区间
- [x] 当前片源从续播点进入时预热续播区间
- [x] 下一集预加载已沿用系列级版本偏好
- [x] 当前集与下一集缓存 key 已隔离
- [x] 详情页预热和播放页兜底预热已共享同一缓存状态
- [ ] 限制下一集预加载不要干扰当前集的起播和播放
- [ ] 确认剧集详情页 current / next 两类任务不会互相抢带宽
- [ ] 确认电影详情页不会在用户离开后继续进行无意义下载
- [ ] 确认续播场景命中的是续播点附近真正需要的起播区间

验收标准：

- 三类触发场景都只准备“够起播”的数据
- 当前内容起播优先级始终高于下一集和旧任务

## Phase E：实现手段收口，而不是围着“下载器”打转

目标：把实现手段降级为实现细节，不让它反过来主导产品行为。

- [x] 已有一层共享缓存写入入口
- [ ] 评估 `StreamCacheDownloadService` 是否继续保留为独立服务
- [ ] 如果该抽象继续增加复杂度，则下沉或内聚回 preload / cache proxy 主链路
- [ ] 不再把“是否存在独立下载器”写成任务验收条件
- [ ] 所有后续改动都以“缓存复用 + 生命周期控制”为验收口径

验收标准：

- 文档、代码和调试口径都不再把“下载器存在与否”当成功标准
- 任何实现形态都必须服务于“起播可复用”和“退出即停”

## Phase F：测试与回归

目标：把这件事变成能持续验证的真实能力，而不是“自动化看起来没问题”。

- [x] 单测：`CacheKey` 稳定性
- [x] 单测：磁盘区间写入与读取
- [x] 单测：并发下载去重
- [x] 单测：代理范围请求命中
- [x] 单测：播放请求等待 in-flight 预加载后直接命中缓存
- [x] 集成测试：预加载完成后播放不重复下载前缀
- [x] 集成测试：播放中缺失区间边播边补
- [ ] 手工回归：剧集详情页进入后，当前集点击播放起播明显更快
- [ ] 手工回归：电影详情页进入后，当前电影点击播放起播明显更快
- [ ] 手工回归：下一集预加载后，切到下一集起播明显更快
- [ ] 手工回归：续播进入时，续播点附近等待明显缩短
- [ ] 手工回归：退出播放页后旧预加载停止
- [ ] 手工回归：切到另一部剧 / 电影后旧预加载停止
- [ ] 手工回归：MPV / EXO 双内核方向一致

验收标准：

- 可以明确证明“等预加载完成后再点播放，起播更快”
- 可以明确证明“前缀没有重复跑流量”
- 可以明确证明“离开当前内容后旧任务不再继续缓存”

## 10. 推荐实施顺序

建议按下面顺序推进，不要再被某个单独抽象带偏：

1. 先定义“什么叫预加载成功”
2. 再定位“起播为什么仍然等待”
3. 再补 owner / scope / cancel 机制
4. 再收口“只做够起播的准备”
5. 最后决定 `StreamCacheDownloadService` 是否继续独立存在
6. 手工回归必须穿插进行，不要等最后才看体感

## 11. 第一优先级 TODO

如果现在只能先做一件事，优先做：

- [ ] 把“点击播放后为什么仍然等待”与“为什么退出后还在缓存”这两个问题定位清楚，并让文档、日志、代码都围绕这两个问题收口

这比继续扩展某个实现层更重要，因为它直接决定后续应该优先改：

- 缓存命中

## 12. 完成定义

当且仅当下面这些条件都成立时，才算这轮预加载真正可用：

- [ ] 剧集详情页进入后，当前集点击播放能直接利用缓存起播
- [ ] 电影详情页进入后，当前电影点击播放能直接利用缓存起播
- [ ] 下一集切换时，下一集能直接利用缓存更快起播
- [ ] 续播进入时，续播点附近能直接利用缓存更快起播
- [ ] 等预加载完成后再点播放，不会重复下载起播前缀
- [ ] 退出播放页后，旧片源不会继续缓存
- [ ] 切到另一部剧 / 电影后，旧片源不会继续缓存
- [ ] MPV / EXO 的方向性结论一致

## 13. 最终判断

当前这件事的重点，已经不应该再表述成：

- “要不要一个下载器服务”
- “要不要继续给后台多加点缓存能力”

更准确的说法应该是：

> 我们要把预加载做成一个真正服务于播放起点的能力。  
> 用户在剧集详情页、电影详情页、以及准备看下一集这三个时机里，提前拿到可播放起点；点击播放后直接使用这段缓存开始观看；离开当前内容后，不再相关的后台任务立即停止。

只要没有做到这件事，无论代码里有多少抽象、多少测试、多少缓存层，都还不算真正完成。
